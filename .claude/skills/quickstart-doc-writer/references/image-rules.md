# 圖片規格詳細說明

當 SKILL.md 的圖片決策規則觸發時，參考此文件產出完整的圖片需求。

## 圖片類型定義

### 1. 定位圖（type: where）

**用途**：告訴使用者「按鈕在哪、欄位在哪」

**必備元素**：
- 圈選/框選/箭頭標註目標位置
- 保留足夠上下文（讓人知道在什麼頁面）
- 如果同一畫面有多個步驟，用編號圓點（①②③）

**截圖建議**：
- 不要只截按鈕特寫，要包含周圍環境讓人知道「在哪個頁面」
- 用紅框或箭頭標註目標
- 若路徑很長（如 設定 → 進階 → 權限 → API），可用多張圖或一張圖配編號

### 2. 成功畫面圖（type: success）

**用途**：告訴使用者「成功長什麼樣」

**必備元素**：
- 成功狀態的畫面（必備）
- 圈選或標註「成功判定點」（例如綠色勾勾、Connected 狀態）

**截圖建議**：
- 優先截「狀態指示」區域
- 如有失敗狀態常見，可額外提供失敗對照

### 3. 對照圖（type: compare）

**用途**：並排展示「正確 vs 錯誤」或「成功 vs 失敗」

**必備元素**：
- 至少兩張圖並排
- 明確標示哪個是正確/成功
- 錯誤的那張標註「常見錯誤」或「❌」

**截圖建議**：
- 兩張圖裁切成一致大小
- 用表格或並排方式呈現

**Markdown 範例**：
```markdown
| ✅ 正確 | ❌ 錯誤 |
|--------|--------|
| ![正確接線方式](./images/cable-correct.png) | ![錯誤接線方式](./images/cable-wrong.png) |

*圖：USB 線必須插入標有「Data」的孔位（左），插入「Charge」孔位無法傳輸資料（右）*
```

### 4. 步驟圖（type: step）

**用途**：實體操作的每個關鍵動作

**必備元素**：
- 清楚的方向與位置（看得懂往哪邊轉、往哪邊插）
- 如有多步驟，用編號標示順序
- 標註關鍵零件名稱

**拍攝建議**：
- 拍攝角度要讓人看得出「方向」
- 手或工具入鏡可幫助理解操作方式
- 避免背景雜亂

## 截圖內容規範

### 裁切原則
- 以「能辨識所在頁面」為前提裁切
- 不要只剩按鈕特寫（除非按鈕本身就是全部）
- 保留麵包屑導航或頁面標題

### 解析度
- 文字必須可讀（避免模糊）
- 建議寬度 720-1200px

### 隱私遮罩（必須）
以下資訊必須遮蔽：
- API Key / Token
- Email / 用戶名稱
- 序號 / License Key
- 客戶資料 / 個資

### 標註風格（建議統一）
- **框選**：紅色或橘色方框，線寬 2-3px
- **箭頭**：指向操作目標
- **編號圓點**：①②③，用於同一畫面多步驟

## Markdown 插入格式

### 基本格式（推薦）

```markdown
![替代文字](./images/filename.png)

*圖 1：圖說內容，包含「看哪裡」+「成功判定」*
```

### 使用 figure（網站文件推薦）

```markdown
<figure>
  <img src="./images/filename.png" alt="替代文字" width="720" />
  <figcaption>圖 1：圖說內容</figcaption>
</figure>
```

### 檔案命名規範

格式：`{章節}-{任務}-{元素}-{平台?}.png`

範例：
- `quickstart-login-submit-web.png`
- `settings-api-key-input-ios.png`
- `troubleshooting-401-error-web.png`

## 圖片密度建議

| 章節類型 | 建議圖片數 |
|---------|-----------|
| 快速開始 | 2-4 張 |
| 單個 How-to 任務 | 1-3 張 |
| Troubleshooting 每個症狀 | 0-2 張 |

原則：讓讀者不用滑很久就能找到關鍵圖。
