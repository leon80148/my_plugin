# 測試與迭代改進

## 自動化測試生成

### 單元測試請求
```
以下是函式 {函式名稱} 的代碼，
請為它生成單元測試，用 {測試框架} 框架，涵蓋：
1. 一般輸入（正常情況）
2. 邊界條件（極端值）
3. 異常情況（錯誤輸入）

{代碼片段}
```

### 整合測試設計
```
請為以下系統設計整合測試方案：
- 關鍵元件：{元件列表}
- 主要用例：{用例描述}

列出每個測試的：前置條件、動作步驟、預期結果。
```

### 性能測試規劃
```
這個 {功能描述} 功能對性能有要求，
請建議：
1. 關鍵性能指標（響應時間、記憶體等）
2. 不同負載水平的測試計畫
3. 推薦的壓力測試工具
```

## 質量保障與安全檢查

### 代碼質量審查
```
請審視以下代碼的潛在問題：
- 安全漏洞（SQL Injection, XSS 等）
- 資源洩漏（檔案、連線未關閉）
- 錯誤處理覆蓋
- 可維護性問題

{代碼片段}
```

### 常見安全檢查項
- [ ] 用戶輸入有驗證和清理
- [ ] 使用參數化查詢防止 SQL Injection
- [ ] API 有適當的認證和授權
- [ ] 敏感資料有加密
- [ ] 錯誤訊息不洩露系統資訊

## 資料驅動的迭代

### 指標設計請求
```
對於一款 {產品類型} App，
我們應該追蹤哪些數據來評估產品健康度和用戶黏性？

請分類列出：
1. 總體業務指標
2. 用戶行為指標
3. 產品特定指標
```

### 常見追蹤指標

**總體指標：**
- DAU / MAU（日活/月活用戶）
- 留存率（1日、7日、30日）
- 轉換率（註冊、付費）
- LTV（用戶終身價值）

**行為指標：**
- 平均使用時長
- 功能使用率
- 完成任務次數
- 流失節點分析

### 埋點規劃
```
請幫我規劃 {產品類型} 的事件追蹤：
1. 需要追蹤的關鍵事件
2. 每個事件需要的參數
3. 使用 {分析工具} 的代碼範例
```

## 迭代循環

```
測試 → 學習 → 改進 → 再測試
```

### 版本規劃請求
```
根據以下用戶反饋和數據分析結果，
請幫我制定產品路線圖：
- 版本 1.1（近期 1-3 個月）
- 版本 1.2（中期 4-6 個月）
- 版本 2.0（遠期）

反饋：{反饋摘要}
數據：{數據摘要}
```
